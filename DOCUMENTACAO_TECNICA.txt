================================================================================
  1. VISÃO GERAL DO SISTEMA
================================================================================

O Sistema Antifraude da Unica Promotora é uma aplicação web interna usada pela
"Mesa de Crédito" para gerenciar, analisar e aprovar/reprovar propostas de
crédito consignado vindas de múltiplos bancos e hubs de integração.

PROBLEMA QUE RESOLVE:
  - Propostas de crédito chegam de múltiplos bancos via APIs externas (Hope/Titan
    e Storm Tecnologia).
  - É necessário identificar rapidamente quais propostas têm risco (CPF na
    blacklist, convênio desconhecido, valor acima do limite, etc.).
  - A equipe precisa de uma visão centralizada de todos os status em tempo real.
  - A Mesa de Crédito precisa consultar e operar contratos diretamente nas APIs
    externas (Storm e Hope) sem sair do sistema.

ARQUITETURA GERAL:
  - Backend  : API REST em Python/FastAPI, banco SQLite (dev) ou PostgreSQL (prod).
  - Frontend : Interface React + TypeScript (Vite), comunicação via axios.
  - Integrações externas: API Titan/Hope (banco Hope via Ceoslab) e API Storm
    Tecnologia (hub multibanco: colaboradores, contratos, simulações).

SEPARAÇÃO STORM / HOPE (CRÍTICA):
  Storm é um HUB que roteia contratos de MÚLTIPLOS bancos (BMG, Pan, Prefeituras,
  etc.). O campo "banco" no Storm é sempre dinâmico — extraído do payload.
  Hope é um BANCO ESPECÍFICO, integrado exclusivamente via API Titan (Ceoslab).
  O campo banco="HOPE" é hardcodado APENAS em hope_adapter.py (backend) e
  hopeAdapter.ts (frontend) — os únicos locais válidos do sistema.
  NUNCA usar banco="HOPE" em contexto Storm.

FLUXO DE APROVAÇÃO DE PROPOSTAS:
  1. Propostas entram via sync (POST /titan/sync ou POST /storm/sync) ou
     manualmente (POST /propostas/).
  2. Motor antifraude avalia regras configuradas no banco e calcula score 0-100.
  3. BLOQUEADO (score >= 80 ou regra bloqueante): proposta vai direto para BLOQUEADA.
  4. DEMAIS CASOS: proposta vai para ANALISE_MANUAL — revisão humana obrigatória.
  5. Analista abre o modal "Opções" no dashboard e clica Aprovar ou Bloquear.
  NOTA: O motor NUNCA aprova automaticamente. Toda aprovação final é humana.

--------------------------------------------------------------------------------
  4.2  backend/app/models.py — Modelos (tabelas do banco)
--------------------------------------------------------------------------------

FUNÇÃO: Define as tabelas do banco de dados usando SQLAlchemy ORM.

TABELA: usuarios (users)
  id, username, nome, cargo, email, perfil, senha_hash, token, ativo

TABELA: corretores
  id, nome, cpf (único), grupo_id (FK → grupos, nullable)

TABELA: grupos
  id, nome (único), limite (float — R$ máximo permitido por grupo)

TABELA: convenios
  id, nome (único), banco, ativo, auto_registrado, criado_em

TABELA: blacklist
  id, cpf (único), motivo, criado_em

TABELA: regras_antifraude
  id, nome, tipo (TipoRegra enum), parametros (JSON), peso_score,
  bloqueante (bool), prioridade, ativo

TABELA: propostas
  id (UUID), proposta_id_externo (único — idempotência), cpf_cliente, banco,
  convenio, produto, valor, status (StatusProposta enum), corretor_id (FK),
  score_fraude, resultado_motor, decisao_detalhes (JSON), payload_original (JSON),
  ultimo_erro, tentativas, criado_em, atualizado_em

TABELA: auditoria
  id, proposta_id (FK), evento (TipoEvento enum), dados (JSON),
  usuario, ip_origem, timestamp

TABELA: log_acesso
  id, acao, usuario_id (FK), ip, user_agent, tipo_entidade, entidade_id,
  antes (JSON), depois (JSON), risco, criado_em

ENUMS:
  StatusProposta: ENFILEIRADA | EM_ANALISE | ANALISE_MANUAL | APROVADA |
                  REPROVADA | BLOQUEADA | ENVIADA_BANCO | CONFIRMADA_BANCO | ERRO
  ResultadoMotor: APROVADO | MANUAL | BLOQUEADO
  TipoRegra: BLACKLIST | VALOR_MAXIMO | BANCO_CONVENIO | UF_BLOQUEADA |
             SCORE_RISCO | LIMITE_DIARIO
  TipoEvento: CRIACAO | ENFILEIRAMENTO | INICIO_ANALISE | DECISAO_MOTOR |
              ALTERACAO_MANUAL | REPROCESSAMENTO


--------------------------------------------------------------------------------
  4.3  backend/app/schemas.py — Schemas de validação
--------------------------------------------------------------------------------

SCHEMA DE AUTENTICAÇÃO:
  LoginRequest  : { identificador: str, senha: str }
  TokenResponse : { access_token, username, nome, cargo, perfil, email }

SCHEMA DE PROPOSTA:
  PropostaCreate : { proposta_id_externo, cpf_cliente, banco, convenio?,
                     produto?, nome_cliente?, uf_cliente?, valor,
                     payload_original? }
  PropostaOut    : PropostaCreate + id, status, score_fraude, resultado_motor,
                   decisao_detalhes, tentativas, criado_em, atualizado_em

SCHEMA DE SUMMARY:
  PropostaSummary : { total, enfileiradas, em_analise, aprovadas, reprovadas,
                      bloqueadas, analise_manual, enviadas_banco,
                      confirmadas_banco, erro }

SCHEMA DE DASHBOARD (Mesa de Crédito):
  PropostaDashboardItem : { id, ade, banco, convenio, produto, corretor,
                            corretor_id, valor, status, cpf, nome_cliente,
                            uf_cliente, observacoes, data_importacao,
                            data_atualizacao, data_agendamento, possui_arquivos,
                            score_fraude, resultado_motor, origem, tentativas }
    -> "ade"    = proposta_id_externo (código externo / "ADE").
    -> "origem" = "hope" | "storm" | "manual" (detectado pelo prefixo do ADE).
    -> "observacoes" = ultimo_erro OU motivo_principal OU payload.observacoes.

  PropostasDashboardResponse : { items: list[PropostaDashboardItem],
                                 total, skip, limit }

--------------------------------------------------------------------------------
  4.6  backend/app/core/circuit_breaker.py — Circuit Breaker
--------------------------------------------------------------------------------

ESTADOS:
  CLOSED    → funcionamento normal; todas as chamadas passam.
  OPEN      → após N falhas, rejeita chamadas imediatamente.
  HALF_OPEN → após timeout_recuperacao segundos, testa recuperação.

FUNÇÃO get_breaker(nome):
  -> Instância global compartilhada por nome de serviço.
  -> Uso: get_breaker("storm_api") → sempre retorna a mesma instância.

NOTA: StormAuthError e TitanAuthError NÃO contam como falha no circuit breaker
(são erros de credencial, não de disponibilidade do serviço).

  4.8  backend/app/routers/propostas.py — Propostas
--------------------------------------------------------------------------------

IDEMPOTÊNCIA: POST com mesmo proposta_id_externo retorna o existente (HTTP 200)
em vez de criar duplicata. Trata race condition via IntegrityError.

ENDPOINTS:
  GET  /propostas/summary             Contagem por status (cards KPI)
  GET  /propostas/dashboard           Painel operacional — usa query_dashboard()
    ATENÇÃO: declarado ANTES de /{proposta_id} no router para não ser capturado.
    Filtros: banco, status, cpf, nome, corretor, valor_min, valor_max,
             data_inicio, data_fim
    Ordenação: order_by + order_dir (padrão: criado_em desc)
    Paginação: skip, limit (máx 200)
  GET  /propostas/                    Lista propostas com filtros básicos
  GET  /propostas/{id}                Detalhe de uma proposta
  GET  /propostas/{id}/auditoria      Histórico de eventos da proposta
  GET  /propostas/{id}/debug          Raio-x completo da decisão antifraude
  POST /propostas/                    Cria proposta e enfileira para o motor
  POST /propostas/{id}/aprovar        Aprova manualmente (exige ANALISE_MANUAL)
  POST /propostas/{id}/bloquear       Bloqueia manualmente (qualquer status)
  POST /propostas/{id}/reprocessar    Roda o motor novamente (exige ERRO/BLOQUEADA)
  POST /propostas/reprocessar-aprovadas  Reenfileira TODAS as APROVADA para
    reavaliação (exige perfil admin/gestor). Útil após mudança no motor.

REFATORAÇÃO (v2.4):
  O endpoint /dashboard delega toda a lógica para
  services/propostas_dashboard.query_dashboard() — o router é apenas roteador.
  Removidos _determinar_origem, _normalizar_proposta e _SORT_COLS inline.

--------------------------------------------------------------------------------
  4.12 backend/app/routers/titan.py — Integração Titan + Sync Hope
--------------------------------------------------------------------------------

FUNÇÃO: Proxy para a API Titan (Hope/Ceoslab) com autenticação Bearer,
cache, retry e circuit breaker. Inclui endpoint de sincronização de propostas.

MAPEAMENTO DE ERROS:
  TitanAuthError → HTTP 502  (credencial inválida — não conta no circuit breaker)
  TitanAPIError  → HTTP 503  (serviço externo indisponível)

ENDPOINTS:
  GET  /titan/status                       Ping real com latência e circuit breaker
  GET  /titan/bancos                       Lista bancos da Titan
  GET  /titan/bancos/{banco_id}/produtos   Produtos de um banco
  GET  /titan/bancos/{banco_id}/referencia Referência completa em paralelo

--------------------------------------------------------------------------------
  4.15 backend/app/services/antifraude.py — Motor de Regras Antifraude
--------------------------------------------------------------------------------

FUNÇÃO: Avalia cada proposta contra regras configuráveis no banco.

COMPORTAMENTO PADRÃO (v2.4 — mudança importante):
  O motor NUNCA aprova automaticamente.
  Score >= 80 OU regra bloqueante → BLOQUEADO.
  Todos os demais casos           → ANALISE_MANUAL (revisão humana obrigatória).
  A aprovação final é sempre feita pelo analista via botão "Aprovar" no modal.

THRESHOLDS:
  SCORE_MANUAL   = 40   — score >= 40 → mensagem de risco elevado no motivo
  SCORE_BLOQUEIO = 80   — score >= 80 → BLOQUEADO automaticamente

AUTO-MAPEAMENTO DE CONVÊNIOS:
  Antes de avaliar regras, _auto_mapear_convenio() cria o convênio na tabela
  convenios se não existir (auto_registrado=True). Elimina status NAO_MAPEADA.

TIPOS DE REGRA:
  BLACKLIST      → CPF do cliente na tabela blacklist
  VALOR_MAXIMO   → proposta.valor > params["valor_maximo"]
  BANCO_CONVENIO → combinação banco+convênio na lista params["combinacoes"]
  UF_BLOQUEADA   → proposta.uf_cliente em params["ufs"]
  SCORE_RISCO    → score por múltiplos fatores (ex: valor > 3x média de referência)
  LIMITE_DIARIO  → corretor ultrapassaria volume diário params["limite_valor_diario"]

RESULTADO SALVO NA PROPOSTA:
  score_fraude      → int 0-100
  resultado_motor   → "BLOQUEADO" | "MANUAL" (não mais "APROVADO" automaticamente)
  decisao_detalhes  → JSON com regras disparadas, motivos, flags, score

  4.17 backend/app/services/hope_adapter.py — Adapter Titan/Hope

FUNÇÃO: Converte operações da API Titan para o formato PropostaCreate interno.

REGRA FUNDAMENTAL:

FUNÇÃO PRINCIPAL:

    Retorna None se CPF, valor ou ID estiverem ausentes.

  4.18 backend/app/services/storm_adapter.py — Adapter Storm
--------------------------------------------------------------------------------

FUNÇÃO: Converte contratos da API Storm para o formato PropostaCreate interno.

REGRA FUNDAMENTAL:
  banco = NUNCA hardcodado aqui. O Storm é um hub multibanco — o banco é
  sempre extraído dinamicamente do payload via extrair_banco(raw).
  extrair_banco() NUNCA retorna "HOPE" — esse hardcode pertence apenas ao
  hope_adapter.py.

FUNÇÕES DE EXTRAÇÃO (robustas a múltiplos formatos Storm):
  extrair_banco(raw)         — banco.nome → banco_nome → nm_banco → convenio.banco
  extrair_convenio(raw)      — convenio.nome → orgao.nome → convenio_nome
  extrair_produto(raw)       — operacao.nome → produto → tabela_nome
  extrair_cpf(raw)           — cliente_contrato.cpf → clienteDados.cpf → cliente.cpf
  extrair_nome_cliente(raw)  — múltiplos campos alternativos
  extrair_uf_cliente(raw)    — endereco.uf → cliente.endereco.uf
  extrair_valor(raw)         — valor_operacao → producao_bruta → valor_bruto → valor
  extrair_id_contrato(raw)   — ff → codigo → id → id_contrato

FUNÇÃO PRINCIPAL:
  mapear_contrato(raw: dict) -> dict | None
    Retorna None se CPF, valor ou ID do contrato estiverem ausentes.
    Prefixo: proposta_id_externo = "storm-{ff}"

COMPATÍVEL COM:
  /antifraude/listar_contratos → contratos da fila de antifraude
  /contratos                   → contratos gerais
  /clientes/cpf                → contratos embutidos no cliente

  4.19 backend/app/services/propostas_dashboard.py — Serviço de Dashboard
--------------------------------------------------------------------------------

FUNÇÃO: Centraliza a lógica de query e normalização do endpoint /propostas/dashboard.
Extraído do router para ser testável e reutilizável por qualquer router ou task.

FUNÇÕES:
  determinar_origem(proposta_id_externo: str) -> str
    "titan-*" → "hope" | "storm-*" → "storm" | else → "manual"

  normalizar_proposta(p: Proposta) -> dict

    observacoes = ultimo_erro OU decisao_detalhes.motivo_principal OU payload.obs
    data_agendamento = payload.data_agendamento OU payload.agendamento

    origem = determinar_origem(p.proposta_id_externo)

  query_dashboard(db, *, banco, status, cpf, nome, corretor, valor_min, valor_max,
                  data_inicio, data_fim, order_by, order_dir, skip, limit)
    -> tuple[list[dict], int]
    Executa a query SQLAlchemy com joinedload(Proposta.corretor) para evitar N+1.
    Retorna (items_normalizados, total_sem_paginacao).
    Colunas ordenáveis (_SORT_COLS): criado_em, atualizado_em, valor, status,
                                     banco, nome_cliente.
    limit é capeado em 200 por segurança.

  ┌──────────────────────────────────────────────────────────────┐
  │ 1. Auto-mapeamento de convênio                                │
  │    Se proposta.convenio não existe em convenios →             │
  │    registra automaticamente (auto_registrado=True)            │
  └──────────────────────────────────────────────────────────────┘
               │
  ┌──────────────────────────────────────────────────────────────┐
  │ 2. Carrega regras ativas ordenadas por prioridade             │
  └──────────────────────────────────────────────────────────────┘
               │
  ┌──────────────────────────────────────────────────────────────┐
  │ 3. Para cada regra:                                           │
  │    - Avalia regra específica (BLACKLIST, VALOR_MAXIMO, etc.)  │
  │    - Se disparou E é bloqueante:                              │
  │        → status = BLOQUEADA (retorna imediatamente)           │
  │    - Se disparou E não bloqueante:                            │
  │        → acumula score_contribuicao                           │
  └──────────────────────────────────────────────────────────────┘
               │
  ┌──────────────────────────────────────────────────────────────┐
  │ 4. Decisão por score acumulado                                │
  │    score >= 80  → BLOQUEADA (bloqueio automático)             │
  │    score < 80   → ANALISE_MANUAL (revisão humana obrigatória) │
  │                                                               │
  │    NOTA: O motor NUNCA gera status APROVADA automaticamente.  │
  │    A aprovação é sempre feita pelo analista no modal.         │
  └──────────────────────────────────────────────────────────────┘
               │
  ┌──────────────────────────────────────────────────────────────┐
  │ 5. Analista acessa Dashboard → botão "Opções"                 │
  │    → Modal PropostaDetalheModal                               │
  │    → Clica Aprovar  → POST /propostas/{id}/aprovar            │
  │       ou Bloquear  → POST /propostas/{id}/bloquear            │
  └──────────────────────────────────────────────────────────────┘


================================================================================
  9. STATUS DE PROPOSTAS — TODOS OS ESTADOS POSSÍVEIS
================================================================================

  STATUS              | ORIGEM          | SIGNIFICADO
  --------------------|-----------------|------------------------------------------
  ENFILEIRADA         | Automático      | Recém-criada, aguardando o motor.
  EM_ANALISE          | Automático      | Motor iniciou a avaliação.
  ANALISE_MANUAL      | Motor/Manual    | Requer revisão humana (padrão do motor).
  APROVADA            | Manual (humano) | Analista aprovou. Nunca automático.
  REPROVADA           | Manual (humano) | Analista reprovou definitivamente.
  BLOQUEADA           | Motor/Manual    | Fraude detectada ou bloqueio manual.
  ENVIADA_BANCO       | Manual          | Proposta enviada ao banco parceiro.
  CONFIRMADA_BANCO    | Manual          | Banco confirmou o crédito.
  ERRO                | Automático      | Falha no processamento pelo motor.

TRANSIÇÕES VIA API:
  → aprovar:     ANALISE_MANUAL → APROVADA
  → bloquear:    qualquer → BLOQUEADA
  → reprocessar: ERRO | BLOQUEADA → ENFILEIRADA

  11. COMO RODAR O PROJETO
================================================================================

PRÉ-REQUISITOS: Python 3.11+, Node.js 18+, Git.

  git clone https://github.com/DevCesarUnica/antifraude-unica.git
  cd antifraude-unica

  cd backend
  pip install -r requirements.txt
  # Criar backend/.env com credenciais Storm (obrigatório para página Storm):
  #   STORM_USERNAME=seu_usuario
  #   STORM_PASSWORD=sua_senha
  #   STORM_CLIENT_ID=seu_client_id
  python seed.py
  python -m uvicorn app.main:app --reload --port 8000

FRONTEND (outro terminal):
  cd frontend
  npm install
  npm run dev
  # Acesse: http://localhost:5173  (Login: admin / admin123)

DOCUMENTAÇÃO INTERATIVA:
  http://localhost:8000/docs   (Swagger UI)
  http://localhost:8000/redoc  (ReDoc)

IMPORTAR PROPOSTAS DO HOPE (após configurar TITAN_API_KEY):
  POST http://localhost:8000/titan/sync
  Params opcionais: max_pages, data_inicio, data_fim, status_id

IMPORTAR PROPOSTAS DA STORM:
  POST http://localhost:8000/storm/sync
  Params opcionais: max_paginas, id_banco, id_status, data_inicio, data_fim

REPROCESSAR TODAS AS APROVADAS (após mudança no motor):
  POST http://localhost:8000/propostas/reprocessar-aprovadas
  Requer perfil admin ou gestor.
